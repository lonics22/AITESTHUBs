## 描述

Swagger 文档中的 `securityDefinitions` / `security` 块定义的认证头（如 `api_key` header）在启发式分类阶段被匹配到 `_CONTEXT_REF_PATTERNS`（`^api_key$` 正则），归类为 `context_ref` 而非 `manual`。

## 根因

`_apply_heuristic()` 在 `ai_import_service.py:92` 中 `api_key` 被列在 `_CONTEXT_REF_PATTERNS`：

```python
_CONTEXT_REF_PATTERNS: List[re.Pattern] = [
    re.compile(r'^api_key$', re.I),
    ...
]
```

而认证问题生成（第 473-477 行）只从 **manual** 参数中检测认证头：

```python
auth_param_names = {"authorization", "x-api-key", "api-key", "api_key"}
auth_headers = [
    p for p in manual_params
    if p.get("location") == "header" and p["name"].lower() in auth_param_names
]
```

由于 `api_key` 被归类为 `context_ref` 而非 `manual`，它不会出现在 `manual_params` 中，因此认证问题永远不会被生成。

## 影响

- **无认证问题**：向导页面 Step 3 缺失"认证方式"下拉选择框
- **认证值需手动填写**：用户只能通过通用参数表格填写 API Key 值，没有专用 UI
- **OAuth2 认证被忽略**：Swagger 中的 `petstore_auth`（OAuth2）安全定义完全不被识别，涉及该安全定义的端点不会获得任何认证设置
- **Save 后 auth 不正确**：生成的 `auth.type` 为 `none`（没有 `api_key` 的端点）或 `apikey`（有 `api_key` header 参数的端点从 context_ref 获得值）

## 复现步骤

1. 上传 Petstore swagger.yaml（包含 `api_key` securityDefinition）
2. 进入 Step 3 问题列表
3. 观察只有 3 个问题（url_domain / param_value / env_var），缺少 auth 问题
4. 分步检查：DELETE `/pet/{petId}` 的 `api_key` header 参数被分类为 context_ref 而非 manual

## 预期行为

认证 header 参数应覆盖启发式分类，始终归类为 `manual`，从而触发认证问题的生成。

## 修复建议

1. 在 `_apply_heuristic()` 中，先检测是否为已知认证 header，若是则直接返回 `"manual"`
2. 或在 `analyze_endpoints()` 中，对认证 header 参数强制覆盖分类结果为 `manual`

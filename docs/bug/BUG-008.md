## 描述

两个问题：

### 问题 A：SSE 进度推送永远只推送终态

`/answers/` 端点在同一个同步请求中完成整个 AI 生成过程（包括 LLM 调用）。任务状态从 `waiting_user` 直接变为 `completed` 或 `failed` 后，前端才建立 SSE 连接。SSE 建立后立即读到终态并退出，从未推送过中间进度（如"正在处理第 3/10 个接口"）。

### 问题 B：SSE `logs` 端点的 `while True` 轮询占用 WSGI 工作线程

后端 SSE 实现为同步 `while True: time.sleep(1)` 轮询数据库，在 WSGI 部署下，每个 SSE 连接永久占用一个工作线程。gunicorn 4 workers 只能同时服务 4 个 SSE 客户端。

## 根因

AI 生成任务在 answers 视图中同步执行，SSE 连接在任务完成后才建立。SSE 的架构设计与业务流不匹配。

## 复现步骤

1. 上传文档，进入步骤 4
2. 观察进度条 — 0% 直接跳到 100%
3. 中间无任何进度变化
4. 或查看浏览器 Network → SSE 连接建立时任务已完成

## 预期行为

### 方案 A（推荐）
将生成执行改为异步（Celery 任务），SSE 建连后实时推送进度。

### 方案 B（轻量）
SSE 连接在触发生成前建立，由 SSE 视图触发生成并实时推送。
